# Error Reporting Format Skill

テスト失敗やエラー調査結果を報告する際の標準フォーマット。

## When to Use

- E2Eテスト失敗の調査報告時
- ユニットテスト失敗の調査報告時
- バグ調査・デバッグ結果の報告時
- 障害調査レポート作成時

## 報告構成（必須順序）

### 1. まず結論（高レベル分類）

最初に、エラーの性質を分類して報告する：

| 分類 | 件数 |
|------|------|
| ソースコードのデグレ | X件 |
| テスト実装の修正が必要 | Y件 |
| 環境・設定の問題 | Z件 |

**一文で要約**: 「X件すべてが〇〇でした。△△はありません。」

### 2. 次に修正状況

| 状況 | 件数 |
|------|------|
| 修正して解決 | X件 |
| 未修正（修正漏れ等） | Y件 |

### 3. 最後に詳細

ファイル別、原因別に詳細を記載。

## 悪い例（避けるべき）

```
❌ いきなり詳細から始める
❌ ファイル名や行番号を最初に列挙
❌ 読み手が全体像を把握する前に細部を説明
```

## 良い例

```
✅ まず「デグレ0件、テスト修正14件」と結論
✅ 次に「10件修正済み、4件未修正」と状況
✅ 最後にファイル別の詳細
```

## 原則

1. **結論ファースト**: 読み手が最も知りたいこと（デグレか否か）を最初に
2. **段階的詳細化**: 概要→状況→詳細の順で深掘り
3. **表形式活用**: 件数や分類は表で視覚的に整理
