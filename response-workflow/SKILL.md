# Response Workflow

ユーザー入力に対して適切に対応するためのワークフロー。

## 基本原則

**質問には答える。指示には実行する。それ以上のことはしない。**

## メインフロー

```
ユーザー入力
    │
    ▼
┌─────────────────────┐
│ 単一か複数かを判定   │
└─────────────────────┘
    │
    ├─→ 単一 ─────────────────────────────┐
    │                                      │
    └─→ 複数 → 分解して個別のリストにする    │
                  │                        │
                  ▼                        ▼
         ┌────────────────────────────────────┐
         │  各項目に対して:                    │
         │  ┌─────────────────┐               │
         │  │ 質問か指示かを判断 │               │
         │  └─────────────────┘               │
         │      │                            │
         │      ├─→ 質問 → 回答する           │
         │      │                            │
         │      └─→ 指示 ─┐                  │
         │                │                  │
         │                ▼                  │
         │      ┌───────────────────────┐    │
         │      │ コード/設定の変更を伴うか？ │    │
         │      └───────────────────────┘    │
         │          │                        │
         │          ├─→ Yes → TDDフロー      │
         │          │                        │
         │          └─→ No → 通常実行 → 報告  │
         └────────────────────────────────────┘
                  │
                  ▼
               終了
```

---

## Step 1: 入力の分解

### 複数入力の特徴

| パターン | 例 |
|----------|-----|
| 並列接続 | 「〜と、〜」「〜して、〜して」 |
| 番号付き | 「1. 〜 2. 〜」 |
| 段落分け | 複数の独立した文 |
| 混在 | 「〜は何？あと〜して」 |

### 分解例

```
入力: 「このファイルの構造は？あと、テストを実行して」

分解結果:
  1. 質問: 「このファイルの構造は？」
  2. 指示: 「テストを実行して」
```

```
入力: 「バグ#1と#2を修正して」

分解結果:
  1. 指示: 「バグ#1を修正」
  2. 指示: 「バグ#2を修正」
```

---

## Step 2: 質問/指示の判断

### 質問の特徴

| パターン | 例 |
|----------|-----|
| 疑問形 | 「〜は何？」「〜ですか？」「どこ？」「なぜ？」 |
| 確認要求 | 「〜を確認して」「〜を教えて」「〜を見せて」 |
| 情報要求 | 「〜について」「〜の状態は？」 |

**質問への対応**: 答えるのみ。勝手に次のアクションを起こさない。

### 指示の特徴

| パターン | 例 |
|----------|-----|
| 命令形 | 「〜して」「〜を実行」「〜を作成」 |
| 依頼形 | 「〜してください」「〜をお願い」 |
| 作業要求 | 「〜を修正」「〜を追加」「〜を削除」 |

**指示への対応**: 実行する。完了したら結果を報告する。

---

## Step 3: TDD適用判断

### TDD適用基準

| パターン | 例 | TDD適用 |
|----------|-----|---------|
| バグ修正 | 「バグを直して」「修正して」 | ✅ |
| 機能追加 | 「機能を追加して」「実装して」 | ✅ |
| リファクタリング | 「リファクタして」「改善して」 | ✅ |
| 設定変更 | 「設定を変えて」「値を変更して」 | ✅ |
| 調査・確認 | 「調べて」「確認して」 | ❌ |
| ドキュメントのみ | 「READMEを更新して」 | ❌ |
| 情報表示 | 「一覧を見せて」 | ❌ |

---

## Step 4: TDDフロー

```
┌─────────────────────────────────────────┐
│           TDDフロー（デグレ対策付き）      │
├─────────────────────────────────────────┤
│                                         │
│  1. 🔍 既存テスト確認                    │
│     │  - 関連するテストを特定            │
│     │  - 影響範囲を把握                  │
│     ▼                                   │
│  2. 📊 影響範囲分析                      │
│     │  - 変更対象のシンボルを特定         │
│     │  - 参照元を列挙                    │
│     │  - 影響を受けるテストを特定         │
│     ▼                                   │
│  3. 🔴 RED: テストを準備                 │
│     │  - 新規: 期待動作のテストを書く     │
│     │  - 既存: 必要なら修正              │
│     │  - テスト実行（失敗を確認）         │
│     ▼                                   │
│  4. 🟢 GREEN: 実装/設定を変更            │
│     │  - テストが通る変更を行う          │
│     │  - テスト実行（成功を確認）         │
│     ▼                                   │
│  5. 🔵 REFACTOR: 必要なら改善            │
│     │  - テストが引き続き成功することを確認│
│     ▼                                   │
│  6. ✅ デグレチェック                    │
│     │  - 関連テスト実行                  │
│     │  - 全体テスト実行                  │
│     │  - カバレッジ確認                  │
│     ▼                                   │
│  7. 📋 報告                             │
│                                         │
└─────────────────────────────────────────┘
```

### デグレチェック詳細

#### テスト実行の範囲

| Step | 範囲 | 説明 |
|------|------|------|
| 1 | 直接関連テスト | 変更したファイルのテスト |
| 2 | 依存関連テスト | 変更に依存するモジュールのテスト |
| 3 | 全体テスト | 全テストスイート実行 |

#### 変更前後の比較

| チェック項目 | 確認内容 |
|-------------|---------|
| テスト数 | 減っていないか |
| カバレッジ | 下がっていないか |
| 型チェック | エラーが増えていないか |
| Lint | 警告が増えていないか |

## 禁止事項

| やってはいけないこと | 理由 |
|----------------------|------|
| 質問に答えた後、勝手に作業を始める | ユーザーが求めていない |
| 指示を受ける前に先回りして作業する | ユーザーの意図を無視している |
| 複数の解釈が可能な場合に勝手に判断する | 確認すべき |
| TDD対象なのにテストなしで実装する | デグレの原因になる |
| 関連テストを実行せずに完了報告する | デグレを見逃す |

---

## 曖昧な場合

入力が質問か指示か不明確な場合:

1. **確認する**: 「〜を実行しますか？それとも情報だけ必要ですか？」
2. **最小限の解釈**: 判断できない場合は質問として扱い、回答のみ

## 例

### 質問の例

```
User: このURLのページコンポーネントは何？

Claude: watch.$streamerId.before-entry.tsx です。
（ここで終了。勝手に中身を調べたり、問題を解決しようとしない）
```

### 指示の例（TDD不要）

```
User: 現在のテスト結果を見せて

Claude: テストを実行して結果を表示します。
（調査のみなのでTDD不要）
```

### 指示の例（TDD必要）

```
User: Bug #8 を修正して

判定: 指示 → コード変更あり → TDDフロー

処理:
  1. 🔍 既存テスト確認: test_site_settings.py
  2. 📊 影響範囲分析: site_settings.py → Console.tsx
  3. 🔴 テスト作成/修正: test_admin_and_manager_share_same_setting
  4. 🔴 テスト実行: FAIL確認
  5. 🟢 実装: site_settings.py 修正
  6. 🟢 テスト実行: PASS確認
  7. ✅ デグレチェック: 全12件パス
  8. 📋 報告: 「修正完了、テスト12件パス」
```

### 複数入力の例

```
User: このファイルの構造は？あと、テストを実行して

分解:
  1. 質問: 「このファイルの構造は？」
  2. 指示: 「テストを実行して」

処理:
  1. → 回答: 「構造は〜です」
  2. → 実行: テスト実行 → 報告
```

### 曖昧な例

```
User: 認証周りにバグがありそう

Claude: 調査しますか？それとも現状の情報だけ必要ですか？
（勝手に調査を始めない）
```

## チェックリスト

回答する前に確認:

- [ ] 入力は単一か複数か？
- [ ] これは質問か、指示か？
- [ ] 質問なら、答えだけで十分か？
- [ ] 指示なら、コード/設定変更を伴うか？
- [ ] TDD対象なら、テストを先に書いたか？
- [ ] デグレチェックを行ったか？
- [ ] 余計なことをしようとしていないか？
