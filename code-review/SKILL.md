# Code Review Skill

異なるAIモデルによる独立レビューとクロスレスポンスを組み合わせた、ハイブリッド型コードレビュースキル。

## 設計思想

- **異なるモデルの異なるバイアス**を活用し、単一モデルでは見落とす問題を発見する
- **安易な同意を抑制**するクロスレスポンスで、指摘の信頼度を高める
- **矛盾はAIが裁定せず人間に委ねる**（研究結果: 強いモデル同士のディベートは逆効果の可能性）
- **コスト効率**: ディベートループではなく、独立レビュー + 1回のクロスレスポンスに限定

## Activation Triggers

- コードレビュー依頼: 「レビューして」「コードを見て」「PRをレビュー」
- Keywords: "review", "レビュー", "コードチェック", "コード確認"

## Workflow

```
ユーザー: 「このコードをレビューして」
    |
    ▼
┌─ code-review（このスキル・メインコンテキスト）──────────┐
│                                                        │
│  Phase 1: 対象コードの特定                              │
│     - git diff / ファイル指定 / PR を判定               │
│     - レビュー対象のコードを収集                        │
│     │                                                  │
│  Phase 2: 並列独立レビュー                              │
│     |--- code-reviewer（Opus・隔離）→ レビュー結果A     │
│     |--- code-critic（Codex・隔離）→ レビュー結果B      │
│     │                                                  │
│  Phase 3: 並列クロスレスポンス                           │
│     |--- code-reviewer（Opus・隔離）                    │
│     |    入力: コード + 自分の結果A + 相手の結果B       │
│     |    → Codex指摘への根拠付きレスポンス              │
│     |--- code-critic（Codex・隔離）                     │
│     |    入力: コード + 自分の結果B + 相手の結果A       │
│     |    → Opus指摘への根拠付きレスポンス               │
│     │                                                  │
│  Phase 4: 統合レポート作成（メインコンテキスト）         │
│     - 指摘の突合: 一致 / 片方のみ / 矛盾 を分類        │
│     - レポート生成                                     │
│     │                                                  │
│  Phase 5: ユーザーに報告                                │
└────────────────────────────────────────────────────────┘
```

## Sub-Agents

| エージェント | モデル | 役割 |
|-------------|--------|------|
| `code-reviewer` | Claude Opus | 独立レビュー + クロスレスポンス |
| `code-critic` | Codex (via MCP) | 独立レビュー + クロスレスポンス |

## Phase 詳細

### Phase 1: 対象コードの特定

ユーザーの指示に応じて対象を判定:

| 入力パターン | 対象 |
|-------------|------|
| 「このファイルをレビューして」 | 指定ファイル全体 |
| 「変更をレビューして」 | `git diff` の差分 |
| 「PR #123 をレビューして」 | PR の差分 |
| 「この関数をレビューして」 | 指定シンボル |
| 指定なし | 未コミットの変更（`git diff` + `git diff --staged`） |

### Phase 2: 並列独立レビュー

2つのサブエージェントを**並列で**呼び出す（Agent tool）。

**code-reviewer への指示**:
```
Mode: 独立レビュー
対象コード: [コード内容]
```

**code-critic への指示**:
```
Mode: 独立レビュー
対象コード: [コード内容]
```

### Phase 3: 並列クロスレスポンス

Phase 2 の結果を交換し、2つのサブエージェントを**並列で**呼び出す。

**code-reviewer への指示**:
```
Mode: クロスレスポンス
対象コード: [コード内容]
あなたの独立レビュー結果: [結果A]
相手（Codex）の独立レビュー結果: [結果B]
```

**code-critic への指示**:
```
Mode: クロスレスポンス
対象コード: [コード内容]
あなたの独立レビュー結果: [結果B]
相手（Claude Opus）の独立レビュー結果: [結果A]
```

### Phase 4: 統合レポート作成

メインコンテキストで、4つの成果物（独立レビュー×2 + クロスレスポンス×2）を突合する。

#### 分類ロジック

| 条件 | 分類 | 信頼度 |
|------|------|--------|
| 両者が同じ箇所を指摘 + クロスレスポンスで双方が根拠付き同意 | **Agreed** | 高 |
| 片方のみ指摘 + 相手がクロスレスポンスで根拠付き同意 | **Agreed** | 高 |
| 片方のみ指摘 + 相手がクロスレスポンスで根拠付き否定 | **Single** | 中 |
| 両者が指摘したが解釈が矛盾 | **Disputed** | 要判断 |
| 片方が指摘 + 相手が根拠付きで反論 | **Disputed** | 要判断 |

### Phase 5: レポート出力

## レポートフォーマット

```markdown
## Code Review Report

### Summary
- Agreed（両者一致）: X件
- Single（片方のみ）: Y件
- Disputed（矛盾・要判断）: Z件

---

### Agreed（両者一致・修正推奨）

#### [Critical/Warning/Info] 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **Opus**: 指摘内容と根拠
- **Codex**: 同意の根拠 or 補足

---

### Single（片方のみ指摘・確認推奨）

#### [Critical/Warning/Info] 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **指摘者（Opus/Codex）**: 指摘内容と根拠
- **相手の見解**: 「問題なし」の根拠

---

### Disputed（矛盾・人間の判断が必要）

#### 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **Opus の主張**: 内容と根拠
- **Codex の主張**: 内容と根拠
- → コンテキストに依存するため、人間の判断が必要です。

---

### Review Process
- Opus 独立レビュー: X件指摘
- Codex 独立レビュー: Y件指摘
- クロスレスポンス後: Agreed X件 / Single Y件 / Disputed Z件
```

## Options

| オプション | 説明 | デフォルト |
|-----------|------|-----------|
| `--file <path>` | レビュー対象ファイルを指定 | 未コミット変更 |
| `--diff` | git diff の差分をレビュー | — |
| `--pr <number>` | PR の差分をレビュー | — |
| `--focus <観点>` | 特定の観点に集中（security / performance / design） | 全観点 |

## Critical Rules

1. **矛盾はAIが裁定しない**: Disputed セクションでは両者の主張を並べるのみ
2. **クロスレスポンスは1回のみ**: ディベートループは行わない（コスト効率・精度の観点）
3. **並列実行**: Phase 2 と Phase 3 はそれぞれサブエージェントを並列で呼び出す
4. **根拠必須**: 全ての指摘・レスポンスにコードの具体的な箇所の引用を求める
5. **レポートは省略しない**: 却下された指摘も Single セクションに根拠付きで記載する
